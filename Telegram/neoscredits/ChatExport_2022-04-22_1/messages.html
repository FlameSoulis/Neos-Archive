In reply to this message
also because of this, using native oculus airlink with steamvr games is ineffective because instead of lightweight host app, you get fully fledged desktop OVR running with its compositor, buffering and other stuff doing unnecessary work
09:04
Its like the frame from the game goes into SteamVR server through shared gpu memory,
Then SteamVR compositor adds overlays to it (such as chaperone, menus) and does reprojection if needed.
Then SteamVR passes the picture to currently active HMD driver which is OVR API Layer for steam. Which passes picture to OVR server.
Then OVR server uses its own compositor which adds its own overlays (such as menus, guardian) and does reprojection if needed.
Then OVR sends picture to quest over network.
Quest does its own reprojection and overlays its own menus and guardian.
And only then picture is displayed.
09:04
Some passes like chaperone or desktop guardian are probably disabled in this pipeline but I doubt it is optimized enough
09:07
airlink does some tricks tho, instead of applying 2d overlays on desktop, it actually streams content of 2d overlays to quest and applies them there.
09:11
Would be much better if in all this setup both SteamVR and OVR compositors removed and all compositing will only be handled by remote HMD software.
